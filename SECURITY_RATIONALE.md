# Обоснование выбора трёх способов повышения безопасности

## Контекст проекта
- Мобильное приложение (Android) с аутентификацией через Firebase
- Хранит личные данные пользователя (гардероб, геолокация)
- Многопользовательское — каждый пользователь должен видеть только свои данные

---

## Выбранные способы и их обоснование

### 🥇 Способ 1: **ШИФРОВАНИЕ ДАННЫХ** ✅ РЕАЛИЗОВАНО

**Почему именно?**
1. **Критичность:** Для любого мобильного приложения это базовый требование безопасности
   - Root-пользователь может извлечь БД и SharedPreferences
   - Без шифрования userId/email хранятся в открытом виде
   - Регулятор (GDPR, CCPA) требует защиты ПДн

2. **Практическая ценность:** 
   - Защищает от утечек при краже устройства
   - Защищает от анализа приложения через logcat/debugger
   - Усложняет жизнь даже опытным злоумышленникам

3. **Технически просто:**
   - `EncryptedSharedPreferences` из `androidx.security` — стандарт индустрии
   - 2-3 строки кода для шифрования данных
   - Работает прозрачно (как обычные SharedPreferences)

4. **Применимость:** 100% всех проектов нуждаются в этом

---

### 🥈 Способ 2: **УПРАВЛЕНИЕ ДОСТУПОМ** ✅ РЕАЛИЗОВАНО

**Почему именно?**
1. **Критическая уязвимость в текущем коде:**
   ```kotlin
   // УЯЗВИМО: нет проверки, что это мой гардероб
   wardrobeRepository.getItems(userId)  
   // Пользователь может передать чужой userId
   ```
   Любой пользователь может видеть/изменять чужие данные!

2. **Нарушение конфиденциальности:**
   - Это нарушение Закона о защите персональных данных
   - Может привести к судебному разбирательству
   - GDPR: потенциальный штраф 4% от оборота

3. **Просто, но перемешивается с другими:**
   - Не требует новых библиотек
   - Реализуется через проверку в бизнес-логике
   - Действительно защищает, а не "выглядит защищённо"

4. **Только в коде (не в БД):**
   - SQL-инъекции не помогут (проверка перед запросом)
   - Обход на фронтенде невозможен (если бэк API)

---

### 🥉 Способ 3: **ИЗОЛЯЦИЯ КОМПОНЕНТОВ** ✅ РЕАЛИЗОВАНО

**Почему именно?**
1. **Архитектурная основа для двух выше:**
   - EncryptedPrefs доступна только через DI контейнер
   - Access control проверяющий код находится в одном месте
   - Если вдруг нужно добавить новый результат, легко это сделать

2. **Масштабируемость:**
   - Сейчас компоненты связаны (AuthRepository нужна WardrobeRepository)
   - С правильной DI структурой легко добавлять новые хранилища
   - Без этого проект превратится в спагетти-код

3. **Тестируемость:**
   - С DI контейнером можно мокировать зависимости в тестах
   - Можно тестировать WardrobeRepositoryImpl с fake AuthRepository
   - Без этого система тестов быстро падает на коленях

4. **Безопасность архитектуры:**
   - Каждый компонент знает только о своём интерфейсе
   - Нельзя случайно создать WardrobeRepository без проверки доступа
   - Меньше место где может быть ошибка безопасности

## Почему НЕ модульность?

❌ **Модульность (Gradle modules)** отложена, потому что:

- Требует полного рефакторинга структуры папок
- Нужно перемещать 50+ файлов
- Для небольшого проекта дает только ~5% выигрыша в безопасности
- Три выше выбранных способа дают 95% эффекта раньше

**Если проект вырастет — модульность добавим позже.**
